## 写在前面

最近在给项目接各种外部工具（GitHub、数据库、监控系统），每接一个都要写一堆对接代码，换个 Agent 框架又得重写。正好看到 MCP 这个协议在圈子里讨论得挺多，就花时间研究了一下。

一句话说清楚 MCP 和 Function Calling 的区别：**Function Calling 是模型怎么"说"我要调工具，MCP 是工具怎么"标准化地被接入"。** 两个不在一个层面，但经常被搞混。

---

## 为什么需要 MCP？

在 MCP 之前，接工具有多痛苦？拿接 GitHub 举例：

你得手写 API 调用代码、处理 OAuth 认证、适配返回格式、把响应转成模型能理解的格式。好不容易接好了，Claude 升个版接口变了，你得改。同时接了十个工具，每个都一套自己的对接代码。然后产品方说这套工具也要给 Cursor 用——重写一遍，因为 Cursor 和 Claude Desktop 的接入方式完全不同。

**碎片化、难复用、强绑定。** 这就是 MCP 出现之前的真实状态。

MCP 的思路很直觉：定一套标准接口。工具提供方按协议实现一个 Server，任何支持 MCP 的 AI 客户端直接连进来，一次实现到处复用。

这个类比已经被说烂了但确实精准：**USB 接口。** USB 之前鼠标用这个口、键盘用那个口、打印机又是另一个。USB 之后所有外设统一接口，厂商只做一次适配，全球所有电脑都能用。

## Client-Server 架构

```
┌─────────────────────────────────────────────┐
│              MCP Client (AI 应用侧)          │
│   Claude Desktop / Cursor / Agent 框架       │
└──────┬──────────┬──────────┬────────────────┘
       │          │          │
       ▼          ▼          ▼
┌──────────┐ ┌──────────┐ ┌──────────┐
│ GitHub   │ │ PostgreSQL│ │ 文件系统  │
│ MCP      │ │ MCP      │ │ MCP      │
│ Server   │ │ Server   │ │ Server   │
└──────────┘ └──────────┘ └──────────┘
```

- **Server** 是工具实现方，封装好各种操作
- **Client** 是 AI 应用侧，连上 Server 自动获得工具能力
- 一个 Client 可以同时连多个 Server

配置方式极简：在 JSON 文件里加几行，重启后 Claude 自动发现并使用。

## 三类核心能力

MCP Server 暴露三类能力，定位不同：

**Tools（工具）— 有副作用的操作**

创建文件、提交代码、发消息、调第三方 API。执行后会改变外部状态，往往不可逆。所以 Tools 通常需要用户授权确认。

**Resources（资源）— 只读数据**

读日志、查数据库、获取文档。不会改变任何东西，只是把数据给模型看。因为无副作用，可以更宽松地暴露给模型。

**Prompts（提示模板）— 可复用的提示词**

预定义的提示词模板，带参数占位符。把团队积累的优质 Prompt 封装成 MCP Prompts，所有人都能复用，不用每次从头写。

我个人觉得 Resources 是最容易被忽视但最有潜力的一个。很多场景下模型只是需要"看一眼"数据，不需要"动手改"。Resources 的只读特性让它可以放心地暴露给模型，不需要走 Tools 那套授权流程，这在 Agent 自动化场景下很有价值。

## 底层通信：JSON-RPC 2.0

核心很简洁：客户端发 JSON 请求（调哪个方法、参数是什么），服务端返回 JSON 响应（结果或错误）。

传输层支持两种方式：

**stdio（本地）：** Server 作为子进程运行，通过管道通信。启动快、延迟低、不需要网络。Claude Desktop 接本地 Server 用的就是这个。

**Streamable HTTP（远程）：** Server 部署在远程，Client 通过 HTTP 连接。适合多个 Client 共享同一个 Server 的场景。

> 早期版本用的是「HTTP + SSE」双端点方案，2025 年 3 月改成了 Streamable HTTP——合并成一个 `/mcp` 端点，更简洁，serverless 也能跑。

## 我的思考

MCP 解决的是**工具接入的标准化问题**，这个问题在 Agent 时代变得特别突出。

以前做单 Agent 应用，接三五个工具写点胶水代码忍忍就过去了。但现在做 Multi-Agent 系统，每个 Agent 可能需要不同的工具集，如果每个工具都要写一套对接代码，工程量是指数级增长的。

MCP 让工具变成了"即插即用"的东西。一个 Agent 需要什么工具，在配置里声明一下就行。这让 Agent 的工具集可以动态组合，而不是硬编码。

另一个我觉得很重要的点是 **MCP 和 Skill 的关系**。Skill 定义的是"怎么用工具"（业务逻辑），MCP 定义的是"工具怎么接进来"（协议标准）。两者配合使用：MCP 负责底层连接，Skill 负责上层编排。

当然 MCP 也有局限。目前生态主要集中在开发工具领域（GitHub、数据库、文件系统），业务工具（ERP、CRM、支付系统）的 MCP Server 还很少。这需要时间。
