## 写在前面

最近看到两篇讲多 Agent 集群的文章，一篇聊"什么时候该拆成多 Agent"，一篇聊"拆了之后怎么协作"。两篇是同一个作者，放在一起读挺有启发，就整理成了这篇笔记。

说实话，最打动我的是作者提到自己早期把一个集群从 9 个 Agent 砍到 5 个的经历。架构图看着挺像那么回事，但跑了一次完整任务就发现 Token 消耗暴涨、延迟严重增加。最后砍掉多余的节点，性能立刻恢复正常。

这个坑我也踩过。所以这篇文章与其说是"多 Agent 架构指南"，不如说是一份**踩坑后的决策清单**——什么时候该拆、怎么拆、拆了之后怎么协作、怎么证明拆得值。

---

## 先聊一个根本问题：为什么要拆？

很多人（包括早期的我）看到别人做了个多 Agent 系统，觉得自己的也该有。但大厂的建议出奇一致：Anthropic 说先从简单方案做起，OpenAI 说先把单 Agent 能力榨干，Microsoft 甚至说如果能写成函数就先别用 Agent。

为什么？因为多 Agent 不是免费的升级。

最直接的成本是钱。多 Agent 系统的 Token 消耗可能是单 Agent 的好几倍，日均万级请求的场景下，这是从每天几百块到几千块的差距。作者说他早期 9 个 Agent 的时候亲眼见过账单翻了几番，这个体感非常具体。

然后是延迟。原本一次推理能完成的事，拆成串行或并行调用多个 Agent，端到端延迟成倍增长。还有调试——单 Agent 出问题看一条推理链就行，多 Agent 出问题要在好几个 Agent 之间排查边界、通信格式、状态一致性，定位时间直接翻倍。

所以结论是：**默认从单 Agent 起步，只有观测到明确信号时才拆。**

---

## 什么时候"不得不拆"？五个信号

拆分决策不应该基于"业务听起来很复杂"，而应该基于可观测的生产指标。作者总结了五个信号，我觉得每个都值得认真看。

**信号一：提示词和工具超载**

这个我自己遇到过。一个 Agent 绑了太多工具，结果处理任务时频繁选错工具，准确率断崖式下降。后来精简了工具集才恢复。

有个很实用的评估：工具数超过 8 个、提示词超过 800 词、业务域超过 3 个、错误率超过 15%——四项里满足两项，就该考虑拆了。道理很简单，提示词太长模型会忽略关键约束，工具太多模型会选错。

**信号二：领域冲突**

这个最容易被忽视。当一个 Agent 对同一实体需要应用相互矛盾的规则时，单 Agent 必然翻车。

比如 Agent 既要负责销售又要负责定价。销售逻辑想给客户折扣，定价逻辑禁止突破底价。两个规则写在同一段提示词里，模型到底听谁的？这不是提示词写得不好，而是两个业务域的治理逻辑天然对立。

**信号三：治理与审计隔离**

如果你在金融、医疗、供应链这些受监管的领域，需要证明"这个决策是怎么做出来的"，单 Agent 的黑箱推理链会让你很痛苦。多 Agent 的好处是每个 Agent 有明确的输入、责任和输出，审计 trail 是结构化的，出了问题能追溯到具体环节。

**信号四：并行化瓶颈**

任务天然可以分成多个独立子任务时，单 Agent 的顺序执行就是延迟瓶颈。但这里有个关键前提——子任务之间真的有独立分支，不是伪并行。把本可以顺序完成的逻辑硬拆成多 Agent 并行调用，反而会因为协调开销更慢。

**信号五：错误率持续高于 15% 且集中在某个环节**

如果错误总是集中在某个特定环节（比如总是搞错合规检查），说明这个环节需要独立的领域上下文和工具集，正是拆分的最佳切入点。

## 怎么拆？别一上来就过度设计

很多团队一拆就过度设计：规划 Agent、检索 Agent、执行 Agent、审稿 Agent、路由 Agent 全上。架构图很酷，系统跑不动。

作者推荐的是渐进式拆分，我觉得这个思路最靠谱：

**第一步：先把单 Agent 做到极致。** 配上好的工具和 RAG，把 80% 的场景跑通，建立基线指标——延迟多少、成本多少、准确率多少。这个阶段不要想拆分的事。

**第二步：第一个拆出来的永远是 Reviewer。** 这是风险最低、收益最明确的拆分。Reviewer 不改变原有执行逻辑，只是在最后加一道关卡。即使 Reviewer 出问题，也不会影响主流程。

**第三步：再抽 Planner。** 当任务类型多样化，不同任务需要完全不同的执行路径时，引入一个轻量级的 Router 做意图识别和任务分发。

**第四步：领域特化。** 当某个执行 Agent 自身也开始超载时，再进一步细分。但每次拆分前要问一个问题：这个新 Agent 是否有不同的 schema、不同的工具集、不同的治理规则？如果只是提示词不同，就别拆。

**提示词差异不等于架构差异。** 这句话我觉得是整篇文章最值钱的一句。

## 拆了之后怎么证明值？六个维度

拆了多 Agent 必须用数据证明它真的优于单 Agent，而不是"架构图更复杂、PPT 更好看"。

最直接的对比是延迟和成本。同一个请求各跑 100 次取 P95 延迟，精确统计每个 Agent 的 Token 消耗。如果多 Agent 让大多数用户等得更久、花更多钱，拆分就没有意义。

然后是准确率。构建 100 个以上 case 的 eval set，覆盖各业务场景。准确率不升反降是最直接的拆分失败信号。

还有调试效率、治理清晰度、可扩展性。作者特别提到 Google Research 的一个发现：多 Agent 在顺序推理任务上有 39%~70% 的性能下降风险。如果你的任务本质是"步骤 A 必须严格先于步骤 B"，拆成多 Agent 只会增加协调开销。

有个盲评机制我觉得很实用：对同一批请求，分别用单 Agent 和多 Agent 处理，隐去架构信息让业务专家评分。如果专家无法稳定区分，说明拆分没有带来实质价值，只是"架构虚荣"。

## 拆了之后怎么协作？四种模式

好了，假设你已经确认"该拆"，接下来的问题是：多个 Agent 放在一起，怎么配合？

作者总结了四种协作模式，我觉得可以理解为一个演进阶梯：从最基础的"谁来干"，到"怎么干"，到"干得对不对"，到"不知道怎么干"。

### 模式一：路由——谁来干

路由模式是最基础的。核心思想就一句话：**这个活交给谁最合适？**

就像医院的分诊台。分诊护士不治病，只做一件事——询问症状，判断科室，把患者分给对应的专科医生。遇到疑难杂症，护士会同时通知多个科室联合会诊。

在 Agent 集群里，路由器做的是同样的事。它不是让所有 Agent 一起去抢同一个任务，也不是让一个 Agent 硬扛所有问题，而是在任务入口设置一道"匹配层"。

架构上是星型网络：路由器在中间，周围是各个专家 Agent。每个专家可以使用不同的模型、不同的提示词、不同的知识库——正是因为存在差异，匹配才有意义。

路由器的核心能力是判断。两种技术路线：一种是嵌入相似度匹配，把任务描述和专家能力描述分别转成向量算相似度；另一种是意图分类模型，直接输出专家标签。工程上常把两者级联使用。

专家之间怎么沟通？靠 A2A 协议——Agent-to-Agent。这相当于多智能体世界的 TCP/IP，定义了 Agent 之间怎么互相发现、声明能力、建立连接、传递信息。

如果单个专家判断可能有盲区，可以把同一问题并行发给多个专家，再用聚合器综合分析。这叫 MoA（Mixture of Agents）。但要注意，不是所有场景都值得付出额外的 Token 和等待时间。

### 模式二：委托——怎么干

路由解决了"交给谁"，但分配正确只是起点。任务交出去之后，**怎么保证它被完整、高质量地交付？**

委托模式的核心原则是意图委派：主节点只定义最终目标和验收标准，不干预子节点的执行路径。怎么干，是接受委托那一方自己决定的。

架构上是有向树：指令自上而下传递，汇报自下而上汇聚。经理 Agent 有全局视角，负责拆解任务、分配、验收。员工 Agent 只关注自己分到的具体任务，不需要了解整体目标。员工之间一般不直接通信——这是刻意为之，减少横向通信避免"消息风暴"。

委托模式处理复杂任务的底层逻辑是递归分解。经理拆成子任务分配给员工，员工发现某个子任务还是太复杂就继续拆，直到每个子任务足够简单。

还有一个关键机制是中断。被动中断是设定风险阈值，超过自动挂起。主动中断是任务切成多个阶段，每阶段完成后等确认。没有中断机制的集群，一旦某个 Agent 跑偏，往往要到最终输出才能发现——中间的 Token 和时间全浪费了。

### 模式三：辩论——干得对不对

单体 Agent 对自己的错误是"盲"的。幻觉和逻辑漏洞在个体视角下是自洽的，就像一个自己改自己作文的人，很难发现所有毛病。

辩论模式的思路：让多个 Agent 围绕同一个问题进行对抗，在互相质疑中暴露各自的认知盲区。

需要三个角色：正方生成初始方案并维护论证；反方系统性地寻找逻辑漏洞和事实错误；法官主持辩论流程并做最终裁决。法官不能用和正反方同等能力的模型，必须用推理能力更强的——否则面对双方各执一词，它没能力做出高质量判断。

为什么有效？两个原因。一是对抗暴露盲区——正方天然倾向找支持自己的证据，反方强制系统面对"如果这个假设不成立呢"。二是多轮交互强制深度思考——每一轮发言都必须回应对手的上一轮论证，不能前后矛盾。

工程上要注意：角色设定要明确立场（不是简单说"你是正方"），多轮辩论要设硬性上限（通常 3~5 轮），法官的模型选择是胜负手。

### 模式四：群体——不知道怎么干

前三种模式都有个前提：系统里有个知道全局、能做出判断的中心节点。但如果连中心节点都不知道答案呢？

群体模式彻底放弃中心调度，让 Agent 在去中心化的交互中自发找到协作方式。就像鸟群——没有一只鸟知道整体编队是什么样的，但通过简单的局部规则（保持距离、对齐方向、避免碰撞），整个群体展现出复杂的协调行为。

这种模式最适合任务本身模糊、无法预设协作流程的场景。

---

## 四种模式的关系

四种模式不是四个并列的选项，而是一条演进之路：路由解决"谁来干"，委托解决"怎么干"，辩论解决"干得对不对"，群体解决"不知道怎么干"。在实践中，它们往往组合使用。

---

## 我的几点收获

第一，单 Agent 是默认答案，多 Agent 是特定约束下的衍生答案。不要为了拆而拆。

第二，第一个拆出来的永远是 Reviewer，风险最低收益最明确。

第三，提示词差异不等于架构差异。别因为提示词不同就拆新 Agent。

第四，每次拆分后必须用数据证明它真的优于单 Agent。盲评机制很实用。

第五，能用一个 Agent 解决的问题，绝不用两个；能用代码解决的问题，绝不用 Agent。这不是保守，是工程经济学。
