## 写在前面

最近读到两篇关于多 Agent 集群架构的姊妹篇，一篇讲"什么时候该拆"，一篇讲"拆了怎么协作"。结合自己的实践，整理成这篇笔记。

核心观点：
- **多 Agent 不是单 Agent 的升级版，而是单 Agent 在特定约束下的"逃生舱"**
- **能用一个 Agent 解决的问题，绝不用两个；能用代码解决的问题，绝不用 Agent**
- **四种协作模式——路由、委托、辩论、群体——是一条从"能用"到"好用"到"可靠"到"智能"的演进之路**

# 上篇：什么时候该拆成多 Agent？

## 一、为什么大厂都在劝"先单后多"？

过去一年，主流厂商的建议出奇一致：

- **Anthropic**：先从简单方案做起
- **OpenAI**：先把单 Agent 能力榨干
- **Microsoft**：如果能写成函数，就先别用 AI Agent
- **Google ADK**：把 Sequential、Parallel 这类 workflow 和 multi-agent 并列为同级选项，而非默认更高级

背后的逻辑不是"单 Agent 技术上限低"，而是**工程经济学**：

| 隐性成本 | 影响 |
|---------|------|
| 协调开销 | Agent 间通信、状态同步、错误传播，每层增加系统复杂度 |
| 延迟叠加 | 原本一次推理能完成的事，拆成多 Agent 后端到端延迟成倍增长 |
| Token 膨胀 | 多 Agent 的 Token 消耗可能是单 Agent 的数倍甚至数十倍 |
| 调试复杂度 | 单 Agent 出问题看一条推理链，多 Agent 要在多个 Agent 间排查 |

**默认从单 Agent 起步，只有在观测到明确的"拆分信号"时才演进，才是性价比最高的策略。**

---

## 二、五个核心信号：什么时候"不得不拆"？

拆分决策不应基于"业务听起来很复杂"，而应基于**可观测的生产指标**。

### 信号 1：提示词与工具超载

- 系统提示词超过 800 词，指令之间开始相互竞争注意力
- 可用工具超过 8~10 个，工具选择准确率显著下降
- 单次推理需要跨越 3 个以上业务域，不同域的规则在同一个上下文里打架

**实用评估函数：** 工具数 > 8、提示词 > 800 词、领域数 > 3、错误率 > 15%，四项中满足两项，就该考虑拆分。

### 信号 2：领域冲突

当 Agent 在不同业务域中对同一实体需要应用**相互矛盾的规则**时，单 Agent 必然翻车。

> 例：Agent 既要负责销售又要负责定价。销售逻辑想给折扣，定价逻辑禁止突破底价。两个规则写在同一段提示词里，模型到底该听谁的？

**这是从单 Agent 向多 Agent 演进的最强信号。**

### 信号 3：治理与审计隔离

金融、医疗、供应链等受监管领域，需要证明"这个决策是怎么做出来的"。单 Agent 的"黑箱推理链"无法提供结构化审计日志。

多 Agent 的优势：每个 Agent 有明确的输入、责任和输出，审计 trail 结构化且可查询。

### 信号 4：并行化瓶颈

任务天然可分解为多个独立子任务时，单 Agent 的顺序执行会成为延迟瓶颈。

关键前提：**子任务之间真的有独立分支，而非伪并行。** 把本可以顺序完成的逻辑硬拆成多 Agent 并行调用，反而会因为协调开销导致更慢。

### 信号 5：错误率持续高于 15% 且呈现领域聚集

错误集中在某个特定环节（比如总是搞错合规检查），说明该环节需要独立的领域上下文和工具集。

| 信号 | 可观测指标 | 触发阈值 | 动作 |
|------|-----------|---------|------|
| 工具超载 | 工具数+提示词长度+错误率 | 工具>8 或 提示词>800词 且 错误率>15% | 先优化单Agent，再考虑拆分 |
| 领域冲突 | 同一实体在不同业务域的规则冲突次数 | 周冲突>3次 | 拆分 |
| 审计隔离 | 监管问询响应时间 | >24小时无法提供完整审计链路 | 拆分 |
| 并行瓶颈 | 子任务间是否有独立分支 | 有3个以上可并行的独立子任务 | 可考虑拆分 |
| 错误率聚集 | 错误率>15% 且集中在特定环节 | 单环节错误率占比>50% | 拆分该环节 |

## 三、最小可行演进路线：怎么拆？

**不要一拆就过度设计。** 推荐「Triage + 渐进抽离」策略，分四个阶段演进：

### 阶段 1：单 Agent 攻坚

一个 Agent，配上优质工具和 RAG。目标是把 80% 的场景跑通，建立清晰的基线指标（延迟、成本、准确率）。这个阶段不要想拆分。

### 阶段 2：先抽 Reviewer

第一个拆出来的永远是**"审查者"而非"执行者"**。因为这是风险最低、收益最明确的拆分——Reviewer 不改变原有执行逻辑，只是在最后加一道关卡。

### 阶段 3：再抽 Planner

当任务类型多样化，且不同任务需要完全不同的执行路径时，引入轻量级的 Planner 或 Router。系统变成：Planner → Specialist Executor → Reviewer。

### 阶段 4：领域特化

当某个 Specialist Executor 自身也开始工具超载或领域冲突时，再将其进一步细分。

**关键原则：** 每次拆分前必须回答——这个新 Agent 是否有不同的 schema、不同的工具集、不同的治理规则？**提示词差异不等于架构差异。**

## 四、评测框架：怎么证明拆得值？

拆了多 Agent 后，必须用数据证明它真的优于单 Agent。

| 维度 | 单 Agent 基线 | 多 Agent 目标 | 评测方法 |
|------|-------------|-------------|---------|
| 端到端延迟 | 2~5 秒 | 并行任务显著低于顺序执行；串行恶化不超过 20% | 同一请求各跑 100 次，取 P95 |
| Token 成本 | 单次推理消耗 | 不超过单 Agent 的 2~3 倍 | 精确统计每个 Agent 的 input/output tokens |
| 任务准确率 | 整体准确率 | 分领域错误率显著下降，整体<10% | 100+ case 的 eval set |
| 调试效率 | 定位时间长 | 缩短 50% 以上 | 记录故障排查平均时长 |
| 治理清晰度 | 杂乱的推理 trace | 结构化、可查询的 Agent 级审计日志 | 5 分钟内提取特定决策的完整链路 |
| 可扩展性 | 全量重部署 | 支持独立升级/回滚单个 Agent | Agent 级 A/B 测试 |

**特别警惕：** Google Research 的报告显示，多 Agent 在**顺序推理任务上有 39%~70% 的性能下降风险**。

**盲评机制：** 对同一批复杂请求，分别用单 Agent 和多 Agent 处理，隐去架构信息让业务专家评分。如果专家无法稳定区分，说明拆分没有带来实质价值。

---

## 五、三个常见陷阱

1. **为了拆分而拆分**：拆分的唯一正当理由是可观测指标恶化，而不是"看起来更像那么回事"
2. **把 workflow 伪装成 multi-agent**：workflow 是确定性的步骤编排，multi-agent 是自治实体之间的动态协作
3. **忽视回退能力**：没有回退能力的多 Agent 架构是高风险架构

# 下篇：四种协作模式

## 六、为什么需要多 Agent 集群？

单体 Agent 面对复杂问题时的能力天花板：
1. **认知宽度不够**：上下文窗口和知识储备有限
2. **长任务专注度衰减**：任务越长，后面的输出质量越差
3. **对自身错误的"盲区"无法自查**：幻觉和逻辑漏洞在个体视角下是自洽的

---

## 七、四种协作模式总览

| 要解决的问题 | 选用的模式 | 概述 |
|-------------|-----------|------|
| 任务来了，该交给谁？ | 路由模式 | 按能力匹配，精准分发 |
| 交出去的任务，怎么保证闭环交付？ | 委托模式 | 全权委托，只看最终结果 |
| 交付的结果，怎么保证质量？ | 辩论模式 | 多方对抗，交叉验证 |
| 任务本身就模糊不清，怎么协作？ | 群体模式 | 自发分工，动态涌现 |

---

## 八、路由模式：让最合适的大脑，去处理最匹配的问题

### 8.1 核心思想

路由模式是多 Agent 集群中最基础、也最直观的协作方式。核心思想：**先问一句"这个活交给谁最合适"**。

医院的分诊台已经用了几十年——分诊护士不负责治病，只做一件事：询问症状，判断科室，把患者精准分给对应的专科医生。

### 8.2 架构：星型网络

```
                    ┌─────────────┐
                    │   路由器     │
                    │  (Router)    │
                    └──────┬──────┘
                           │
              ┌────────────┼────────────┐
              │            │            │
        ┌─────┴─────┐ ┌───┴───┐ ┌─────┴─────┐
        │ 专家Agent A│ │专家 B │ │ 专家Agent C│
        │ (模型1)    │ │(模型2)│ │ (模型3)    │
        └───────────┘ └───────┘ └───────────┘
                           │
                    ┌──────┴──────┐
                    │  聚合器      │ (可选)
                    └─────────────┘
```

三个核心角色：
- **路由器**：入口网关，职责是"做出选择"——判断任务属于哪个领域，精准分发
- **专家 Agent**：术业有专攻。每个专家可以使用不同的模型、提示词、知识库
- **聚合器（可选）**：多专家联合会诊时，对各专家输出进行整合

设计哲学是**高度解耦**：新增或移除一个专家节点，不需要修改路由器的调度逻辑。

### 8.3 三个核心机制

**语义路由——如何判断"该把任务分给谁"？**
- **嵌入相似度匹配**：将任务描述和专家能力描述分别转化为向量，计算相似度
- **模型意图分类**：用轻量级分类模型直接做意图识别
- 工程实践中常采用级联组合：先意图分类做初筛，再相似度匹配做精排

**A2A 协议——不同 Agent 之间如何沟通？**

A2A（Agent-to-Agent）协议相当于多智能体世界的"TCP/IP 协议"：
1. **发现**：新 Agent 注册自己的能力和标签
2. **能力声明**：用 JSON Schema 描述能处理的任务类型
3. **握手**：建立通信链路，确认协议版本和数据格式
4. **响应**：按约定格式返回结果

**Agent 混合聚合（MoA）——如何综合多个专家的意见？**

将同一问题并行分发给多个专家，聚合器综合分析后输出统一结果。
- **加权投票**：按专家历史准确率给权重
- **专门的聚合模型**：对多份输出进行二次推理和矛盾检测

**注意：不是所有场景都值得付出额外的 Token 和等待时间。**

---

## 九、委托模式：把任务交出去，不只是"分活"，更是"托付"

### 9.1 核心问题

路由模式解决了"该交给谁"，但分配正确只是协作的起点。任务交出去之后：**怎么保证它被完整、高质量地闭环交付？**

核心原则：**意图委派**——主节点只定义最终目标和验收标准，不干预子节点的执行路径。

### 9.2 架构：有向树

```
              ┌─────────────┐
              │  经理Agent   │
              │ (全局视角)    │
              └──────┬──────┘
                     │
           ┌─────────┼─────────┐
           │         │         │
     ┌─────┴───┐ ┌───┴───┐ ┌──┴──────┐
     │员工Agent│ │员工    │ │员工Agent│
     └─────────┘ └───────┘ └─────────┘
                     │
              ┌──────┴──────┐
              │  人在回路    │
              │  (HITL)     │
              └─────────────┘
```

### 9.3 三个核心机制

**递归分解**：经理接到复杂任务 → 拆成子任务 → 分配给员工 → 员工发现太复杂 → 继续拆解 → 直到足够简单。

**分发与归约**：适合并行处理大规模数据。经理切分数据 → 分发给多个员工并行处理 → 回收成果统一汇总。

**中断机制**：
- **被动中断**：设定风险阈值，超过自动挂起
- **主动中断**：任务切成多个阶段，每阶段完成后等待确认

**没有中断机制的集群，一旦某个 Agent 跑偏，往往要到最终输出阶段才能发现——整个中间过程的 Token 和时间全浪费了。**

## 十、辩论模式：让 Agent 在对抗中逼近真相

### 10.1 核心问题

单体 Agent 对自己的错误是"盲"的。幻觉和逻辑漏洞在个体视角下往往是自洽的——就像一个自己改自己作文的人，很难发现所有毛病。

### 10.2 架构：三方对抗

```
     ┌───────────┐         ┌───────────┐
     │  正方      │ ◄─────► │  反方      │
     │ (提案方)   │  对抗   │ (挑战方)   │
     └─────┬─────┘         └─────┬─────┘
           │                     │
           └──────────┬──────────┘
                      │
               ┌──────┴──────┐
               │  法官        │
               │ (仲裁方)     │
               └─────────────┘
```

三个角色：
- **正方**：生成初始方案，提供支撑性证据。建设性、防御性
- **反方**：系统性审视正方的每个论证环节，寻找逻辑漏洞。批判性、攻击性
- **法官**：主持辩论流程，汇总论点，做出裁决。**必须使用推理能力更强的模型**

### 10.3 为什么有效？

**对抗暴露盲区**：正方天然倾向寻找支持自己的证据。反方的存在，强制系统面对"如果这个假设不成立呢？"

**多轮交互强制深度思考**：每一轮发言都必须直接回应对手的上一轮论证。你上一轮说过的话，对方会拿放大镜看，不能前后矛盾。

### 10.4 工程细节

1. **角色设定要明确立场**：不是简单说"你是正方"，而是给出具体的角色身份和能力边界
2. **多轮辩论要设硬性上限**：通常 3~5 轮，同时设定终止条件
3. **法官的模型选择是胜负手**：正反方可以用能力中等的模型形成对抗，但法官必须用推理能力更强的模型

---

## 十一、群体模式：没有指挥中心，协作却自然发生

### 11.1 核心问题

前三种模式都有一个前提：系统里有一个知道全局、能做出判断的中心节点。

但如果连中心节点都无法做出判断呢？有些任务在开始之前，没有人知道正确的协作方式是什么。

### 11.2 核心思想

群体模式彻底放弃中心调度，让 Agent 在去中心化的交互中，自发找到协作方式。

就像鸟群飞行：没有一只鸟知道整体的飞行编队是什么样的，但通过简单的局部规则（保持距离、对齐方向、避免碰撞），整个群体展现出了复杂的协调行为。

### 11.3 关键机制

- **局部感知**：每个 Agent 只能看到邻居的状态
- **简单规则**：基于局部信息做决策
- **涌现行为**：全局协作从局部交互中自发产生

---

## 十二、四种模式的关系

```
能用 ──────► 好用 ──────► 可靠 ──────► 智能
路由模式     委托模式     辩论模式     群体模式
(分发)      (执行)      (验证)      (涌现)
```

在实践中，它们往往组合使用：路由分发任务 → 委托确保执行 → 辩论验证结果。

---

## 总结

1. **单 Agent 是默认答案，多 Agent 是特定约束下的衍生答案**
2. **拆分的唯一正当理由是可观测指标恶化**
3. **第一个拆出来的永远是 Reviewer，风险最低收益最明确**
4. **每次拆分后必须用数据证明它真的优于单 Agent**
5. **能用一个 Agent 解决的问题，绝不用两个；能用代码解决的问题，绝不用 Agent**
