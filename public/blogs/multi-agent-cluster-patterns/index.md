## 写在前面

最近读到一篇关于多 Agent 集群架构的文章，把四种核心协作模式讲得很透彻。结合自己的实践，整理成这篇笔记。

核心观点：**单体 Agent 不够用，但多 Agent 放一起也不是随便放的。** 四种协作模式——路由、委托、辩论、群体——是一条从"能用"到"好用"到"可靠"到"智能"的演进之路。

---

## 为什么需要多 Agent 集群？

单体 Agent 在处理明确、单线程的任务时表现尚可，但面对复杂问题——供应商选择、全链路优化、多维度决策——能力天花板立刻暴露：

1. **认知宽度不够**：一个 Agent 的上下文窗口和知识储备有限
2. **长任务专注度衰减**：任务越长，后面的输出质量越差
3. **对自身错误的"盲区"无法自查**：幻觉和逻辑漏洞在个体视角下是自洽的

**这不是模型能力的问题，是架构的问题。** 一个人再聪明，面对需要多视角交叉验证的复杂决策，也需要团队。

---

## 四种协作模式总览

| 要解决的问题 | 选用的模式 | 概述 |
|-------------|-----------|------|
| 任务来了，该交给谁？ | 路由模式 | 按能力匹配，精准分发 |
| 交出去的任务，怎么保证闭环交付？ | 委托模式 | 全权委托，只看最终结果 |
| 交付的结果，怎么保证质量？ | 辩论模式 | 多方对抗，交叉验证 |
| 任务本身就模糊不清，怎么协作？ | 群体模式 | 自发分工，动态涌现 |

---

## 一、路由模式：让最合适的大脑，去处理最匹配的问题

### 1.1 核心思想

路由模式是多 Agent 集群中最基础、也最直观的协作方式。核心思想很简单：**先问一句"这个活交给谁最合适"**。

这个逻辑，医院的分诊台已经用了几十年。分诊护士不负责治病，只做一件事——询问症状，判断科室，把患者精准分给对应的专科医生。

### 1.2 架构：星型网络

```
                    ┌─────────────┐
                    │   路由器     │
                    │  (Router)    │
                    └──────┬──────┘
                           │
              ┌────────────┼────────────┐
              │            │            │
        ┌─────┴─────┐ ┌───┴───┐ ┌─────┴─────┐
        │ 专家Agent A│ │专家 B │ │ 专家Agent C│
        │ (模型1)    │ │(模型2)│ │ (模型3)    │
        └───────────┘ └───────┘ └───────────┘
                           │
                    ┌──────┴──────┐
                    │  聚合器      │ (可选)
                    │ (Aggregator) │
                    └─────────────┘
```

三个核心角色：
- **路由器**：系统的入口网关，职责是"做出选择"——判断任务属于哪个领域，然后精准分发
- **专家 Agent**：术业有专攻的执行单元。每个专家可以使用不同的模型、不同的提示词、不同的知识库
- **聚合器（可选）**：多专家联合会诊时，对各专家输出进行整合

设计哲学是**高度解耦**：新增或移除一个专家节点，不需要修改路由器的调度逻辑。

### 1.3 三个核心机制

**机制一：语义路由——如何判断"该把任务分给谁"？**

两种技术路线：
- **嵌入相似度匹配**：将任务描述和专家能力描述分别转化为向量，计算向量相似度
- **模型意图分类**：用轻量级分类模型直接做意图识别，输出专家标签

工程实践中常采用级联组合：先用意图分类做初筛，再用相似度匹配做精排。

**机制二：A2A 协议——不同 Agent 之间如何沟通？**

A2A（Agent-to-Agent）协议相当于多智能体世界的"TCP/IP 协议"，定义了四个步骤：
1. **发现**：新 Agent 注册自己的能力和标签
2. **能力声明**：用 JSON Schema 描述能处理的任务类型
3. **握手**：建立通信链路，确认协议版本和数据格式
4. **响应**：按约定格式返回结果

**机制三：Agent 混合聚合（MoA）——如何综合多个专家的意见？**

将同一问题并行分发给多个专家，聚合器综合分析后输出统一结果。

聚合策略：
- **加权投票**：按专家历史准确率给权重
- **专门的聚合模型**：对多份输出进行二次推理和矛盾检测

注意：**不是所有场景都值得付出额外的 Token 和等待时间。**

## 二、委托模式：把任务交出去，不只是"分活"，更是"托付"

### 2.1 核心问题

路由模式解决了"该交给谁"，但分配正确只是协作的起点。任务交出去之后：**怎么保证它被完整、高质量地闭环交付？**

核心原则：**意图委派**——主节点只定义最终目标和验收标准，不干预子节点的执行路径。

### 2.2 架构：有向树

```
              ┌─────────────┐
              │  经理Agent   │
              │ (全局视角)    │
              └──────┬──────┘
                     │
           ┌─────────┼─────────┐
           │         │         │
     ┌─────┴───┐ ┌───┴───┐ ┌──┴──────┐
     │员工Agent│ │员工    │ │员工Agent│
     │ (子任务1)│ │(子任务2)│ │ (子任务3)│
     └─────────┘ └───────┘ └─────────┘
                     │
              ┌──────┴──────┐
              │  人在回路    │
              │  (HITL)     │
              └─────────────┘
```

关键角色：
- **经理 Agent**：拥有全局视角，负责拆解任务、分配、验收
- **员工 Agent**：只关注自己的具体任务，不需要了解整体目标
- **人在回路（HITL）**：执行前审查任务拆解，执行后评审产出成果

### 2.3 三个核心机制

**机制一：递归分解**

经理接到复杂任务 → 拆成子任务 → 分配给员工 → 员工发现子任务太复杂 → 继续拆解 → 直到每个子任务足够简单。

终止条件：子任务已原子化，或再无可用员工。

**机制二：分发与归约**

适合并行处理大规模数据的场景：
1. 经理将数据按批次切分
2. 分发给多个员工并行处理
3. 回收所有成果，统一汇总

**机制三：中断机制**

- **被动中断**：设定风险阈值，超过自动挂起
- **主动中断**：任务切成多个阶段，每阶段完成后等待确认

**没有中断机制的集群，一旦某个 Agent 跑偏，往往要到最终输出阶段才能发现——整个中间过程的 Token 和时间全浪费了。**

## 三、辩论模式：让 Agent 在对抗中逼近真相

### 3.1 核心问题

单体 Agent 对自己的错误是"盲"的。幻觉和逻辑漏洞在个体视角下往往是自洽的——就像一个自己改自己作文的人，很难发现所有毛病。

### 3.2 架构：三方对抗

```
     ┌───────────┐         ┌───────────┐
     │  正方      │ ◄─────► │  反方      │
     │ (提案方)   │  对抗   │ (挑战方)   │
     └─────┬─────┘         └─────┬─────┘
           │                     │
           └──────────┬──────────┘
                      │
               ┌──────┴──────┐
               │  法官        │
               │ (仲裁方)     │
               └─────────────┘
```

三个角色：
- **正方（提案方）**：生成初始方案，提供支撑性证据。角色定位：建设性、防御性
- **反方（挑战方）**：系统性审视正方的每个论证环节，寻找逻辑漏洞。角色定位：批判性、攻击性
- **法官（仲裁方）**：主持辩论流程，汇总论点，做出裁决。**法官必须使用推理能力更强的模型**

### 3.3 为什么有效？

**对抗暴露盲区**：正方天然倾向寻找支持自己的证据。反方的存在，强制系统面对"如果这个假设不成立呢？"

**多轮交互强制深度思考**：每一轮发言都必须直接回应对手的上一轮论证。你上一轮说过的话，对方会拿放大镜看，不能前后矛盾。

### 3.4 工程细节

1. **角色设定要明确立场**：不是简单说"你是正方"，而是给出具体的角色身份和能力边界
2. **多轮辩论要设硬性上限**：通常 3-5 轮，同时设定终止条件
3. **法官的模型选择是胜负手**：正反方可以用能力中等的模型形成对抗，但法官必须用推理能力更强的模型

## 四、群体模式：没有指挥中心，协作却自然发生

### 4.1 核心问题

前三种模式都有一个前提：系统里有一个知道全局、能做出判断的中心节点。

但如果连中心节点都无法做出判断呢？有些任务在开始之前，没有人知道正确的协作方式是什么。

### 4.2 核心思想

群体模式彻底放弃中心调度，让 Agent 在去中心化的交互中，自发找到协作方式。

就像鸟群飞行：没有一只鸟知道整体的飞行编队是什么样的，但通过简单的局部规则（保持距离、对齐方向、避免碰撞），整个群体展现出了复杂的协调行为。

### 4.3 关键机制

- **局部感知**：每个 Agent 只能看到邻居的状态
- **简单规则**：基于局部信息做决策
- **涌现行为**：全局协作从局部交互中自发产生

### 4.4 适用场景

- 任务本身模糊，无法预设协作流程
- 需要探索性的创意工作
- 需要自适应的动态任务分配

## 五、四种模式的关系

```
能用 ──────► 好用 ──────► 可靠 ──────► 智能
路由模式     委托模式     辩论模式     群体模式
(分发)      (执行)      (验证)      (涌现)
```

这四种模式不是四个并列的选项，而是一条演进之路：

1. **路由模式**：解决"谁来干"——最基础，最先落地
2. **委托模式**：解决"怎么干"——保证执行深度
3. **辩论模式**：解决"干得对不对"——保证质量
4. **群体模式**：解决"不知道怎么干"——自组织

在实践中，它们往往组合使用：路由分发任务 → 委托确保执行 → 辩论验证结果。

## 六、落地建议

### 如果只选一个模式先落地

**选路由模式。** 你只需要做好三件事：
1. 定义好专家的能力边界
2. 设计好路由器的匹配规则
3. 在必要时引入一个简单的聚合器

### 如果要保证执行质量

**加上委托模式。** 关键是：
1. 意图委派，不干预执行路径
2. 递归分解，直到子任务足够简单
3. 人在回路，审查拆解和验收成果

### 如果要保证输出可靠性

**加上辩论模式。** 关键是：
1. 正反方角色明确
2. 法官用更强的模型
3. 设定轮数上限，避免无限循环

### 成本意识

**不是所有场景都需要多 Agent。** 简单任务用单体 Agent 就够了。多 Agent 集群的 Token 消耗和响应延迟都会增加，只在复杂决策场景中才值得投入。
